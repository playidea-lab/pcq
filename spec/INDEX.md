# pcq Spec — Index

> Single source of truth for the pcq contract surface. This directory
> holds the *machine-readable* and *human-readable* parts of pcq's
> agent-operable contract — the layer everyone (CQ Go service,
> third-party clients, conformance suites, agent runtimes) targets.

## Why a separate `spec/` directory

`pcq` (the Python library) implements the contract; `spec/` *is* the
contract. Keeping them apart means:

- Other languages and runtimes can read the contract without depending
  on Python.
- The frozen JSON contract registry in
  [`src/pcq/agent/json_contracts.py`](../src/pcq/agent/json_contracts.py)
  has a single inspectable companion (machine-readable JSON Schemas in
  [`schemas/`](./schemas/), generated by
  [`scripts/export_schemas.py`](../scripts/export_schemas.py)).
- Versioning, deprecation, and conformance can be discussed
  independently from any one implementation's release cycle.

## Where this fits

- `src/pcq/` — Python implementation of the contract.
- `spec/` — the contract itself (this directory).
- `tests/conformance/` *(coming T-PCS-003)* — golden input/output pairs that any conformant implementation must satisfy.
- `docs/` — human-facing guides (`MCP_INTEGRATION.md`, `AGENT_OPERATING_GUIDE.md`, `INTRODUCTION.md`, case studies, …). Spec content is not duplicated here; the old `docs/SPEC.md`, `docs/JSON_CONTRACTS.md`, etc. now redirect to `spec/`.

## Stability and change policy

- The frozen registry in `src/pcq/agent/json_contracts.py` has been
  stable since v2.13. Changes within `schema_version=1` are
  **additive-only**: callers may depend on the required fields, and
  pcq may add more.
- Breaking change → `schema_version` MAJOR bump → see `VERSIONING.md`
  (next PR) for the full rules and timeline.
